## 一、整体技术架构与可行性总览
### 1. 核心架构模式
所有模块统一采用 **「IntelliJ 插件（Kotlin/JVM） + Rust 原生动态库」** 的分层架构：
- **上层插件**：负责 UI 渲染、IDE 扩展点对接、用户交互，复用 IntelliJ 平台的生态能力
- **Rust 核心层**：承担所有计算密集、IO 密集的核心逻辑，编译为对应平台的动态库（`.dylib`/`.so`/`.dll`）
- **交互层**：通过 JNI（Java Native Interface）实现跨语言调用，同进程内交互，延迟最低、开销最小

### 2. 跨语言交互方案选型
| 方案 | 性能 | 开发难度 | 适用场景 |
|------|------|----------|----------|
| JNI（推荐） | 最高，直接内存访问，调用纳秒级开销 | 中等，需定义 native 接口 | 全局搜索、日志解析等高频低延迟场景 |
| JNA | 中等，调用有微秒级开销 | 低，Java 侧无需手写 native 声明 | 构建日志分析等低频调用场景 |
| 独立进程 + IPC | 低，序列化+通信开销大 | 高，需处理进程生命周期 | 不推荐，会抵消 Rust 的性能优势 |

**结论**：核心性能模块统一采用 JNI 方案，Rust 侧通过官方 `jni` crate 实现接口，技术成熟稳定，IntelliJ 生态已有大量插件采用该模式。

### 3. 整体可行性结论
- **完全可行**：5 个模块均可以通过标准插件扩展点实现，无需修改 Android Studio 核心代码，无合规风险。
- **优先级排序**：全局文本搜索 > 高性能 Logcat > Gradle 构建日志分析 > Tree-sitter 语法高亮 > VFS 增量扫描。
- **前置验证**：前 3 个模块边界完全独立，无 IDE 核心耦合，可直接落地开发；后 2 个有一定平台限制，建议先验证前 3 个再推进。

## 二、各模块详细实现方式与可行性验证
### 模块 1：全局文本搜索引擎（Find in Path）- 优先级最高
#### 功能边界与 IDE 集成方案
两种集成路径，MVP 阶段推荐方案 B 快速验证，正式版对接方案 A 实现无缝体验。
- **方案 A：对接原生 Find in Path 扩展点**
  - 扩展点：`com.intellij.search.textSearcher`，实现 `TextSearcher` 接口
  - 优势：用户无感知，复用原生 UI、快捷键、搜索历史、批量替换能力
  - 成本：需适配 IntelliJ 平台搜索 API，不同版本存在少量差异
- **方案 B：独立搜索工具窗口**
  - 自定义 Tool Window，自主实现搜索 UI 与结果展示
  - 优势：完全可控，不依赖 IDE 内部 API，兼容性强，开发周期短
  - 成本：用户需适应新入口，无法直接复用原生替换功能

#### Rust 核心实现方案
- **核心依赖**：直接复用 ripgrep 官方生态库，避免重复造轮子
  - `ignore`：处理 `.gitignore`/`.ignore` 规则、并行文件遍历与过滤
  - `grep-searcher` / `grep-regex` / `grep-matcher`：ripgrep 核心搜索引擎，支持字面量、正则、上下文行提取
  - `jni`：JNI 接口绑定
  - `crossbeam-channel`：并发搜索与流式结果输出
- **核心逻辑**：
  1. 接收 JVM 传入的搜索参数（根目录、匹配模式、过滤规则、上下文行数）
  2. 并行遍历文件，应用忽略规则与文件类型过滤
  3. 流式执行文本匹配，边搜索边返回结果，支持中途取消
  4. 按行号、列号封装匹配结果，返回给 JVM 层

#### 性能收益预期
对比 Android Studio 原生搜索，在 10 万文件级的大型项目中：
- 搜索速度提升 3~8 倍，正则搜索提升更明显
- 搜索过程内存占用降低 70%~85%，无 JVM GC 卡顿
- 冷启动搜索无索引等待，直接遍历文件系统

#### 可行性确认
✅ 100% 可行。IntelliJ 平台官方提供搜索扩展点，社区已有 RipGrep 类插件验证过该技术路径，无技术壁垒。

### 模块 2：高性能流式 Logcat 面板 - 优先级第二
#### 实现思路
完全独立于原生 Logcat，插件自主实现日志面板：通过 ADB 命令获取日志流，Rust 侧负责流式解析、过滤、索引，JVM 侧仅做可视区域的虚拟滚动渲染。

#### Rust 核心能力
1. **流处理**：启动 `adb logcat` 子进程，零拷贝读取标准输出流，逐行解析
2. **结构化解析**：识别日志的时间、PID、TID、日志级别、标签、消息体
3. **高性能过滤**：支持标签、级别、关键词、正则、PID 多条件组合过滤，百万行日志过滤耗时 < 50ms
4. **增量索引**：后台构建倒排索引，支持日志内全文秒搜
5. **环形缓冲**：可配置最大日志行数，自动淘汰旧日志，内存占用稳定

#### 插件 UI 层实现
- 自定义 Tool Window，采用虚拟滚动列表（仅渲染可视区域 50~100 行）
- 过滤栏、级别切换、日志高亮与原生 Logcat 体验对齐
- 支持点击栈追踪跳转到对应代码行

#### 可行性确认
✅ 100% 可行。与 IDE 核心零耦合，仅依赖 ADB 命令行工具，开发门槛最低，体感收益极强。

---

### 模块 3：Tree-sitter 语法高亮增强 - 优先级第三
#### 实现边界
**只替换语法级高亮，不触碰 PSI 语义引擎**。仅负责基于语法结构的 token 高亮，变量类型、引用解析等语义高亮仍保留原生实现，避免破坏重构、跳转等核心功能。

#### 实现方案
- **Rust 侧**：封装 `tree-sitter` 核心库，以及 `tree-sitter-kotlin`/`tree-sitter-java`/`tree-sitter-xml` 等语言解析包，提供增量解析接口：输入文件内容与变更范围，输出 token 类型与位置列表。
- **插件侧**：实现 `com.intellij.lang.syntaxHighlighterFactory` 扩展点，自定义 `SyntaxHighlighter`。编辑器内容变更时，调用 Rust 侧增量解析，将 Tree-sitter 输出的 token 映射为 IntelliJ 的文本属性，实现高亮。

#### 限制与可行性
⚠️ 可行但有明确边界：
- 收益集中在大文件（> 1000 行）滚动、快速输入场景，输入延迟可降到 10ms 以内
- 无法实现语义级高亮（如成员变量、静态方法区分），高亮精细度略逊于原生
- 不影响 IDE 其他功能，风险可控

---

### 模块 4：VFS 增量扫描增强 - 优先级第四
#### 可行边界
**插件无法替换 IntelliJ 原生 VFS 底层实现**，仅能做辅助增强，无法从根源上优化 IDE 启动的 VFS 刷新。

#### 落地方式
- Rust 实现独立的文件系统监听与快速扫描，服务于插件自身的功能（如搜索索引更新、文件变更检测）
- 项目启动时预扫描文件变更，主动触发 IDE VFS 局部刷新，减少全量扫描概率

#### 结论
性价比低，和 IDE 原生能力存在重叠，容易造成重复扫描，建议作为后期可选优化，不纳入初期开发计划。

### 模块 5：Gradle 构建日志分析 - 优先级第五（入门最简单）
#### 实现方案
- 插件监听 Gradle 构建事件，获取构建输出的原始日志
- 传入 Rust 侧流式解析，识别每个任务的起止时间、耗时、状态、错误堆栈
- 输出任务耗时排行、瓶颈模块提示、错误归因，展示在独立工具窗口

#### 可行性确认
✅ 100% 可行，逻辑完全独立，开发难度最低，适合作为 Rust 插件开发的练手项目。

## 三、模块 1（全局文本搜索）详细开发计划与架构预研
### 1. 最终技术栈
| 层级 | 技术选型 |
|------|----------|
| 插件层 | Kotlin + IntelliJ Platform SDK，兼容 Android Studio 2023.1+ |
| 构建工具 | Gradle + IntelliJ Gradle Plugin |
| Rust 层 | Rust 1.75+，依赖 `ignore` `grep-searcher` `jni` `thiserror` |
| 跨平台编译 | Cargo + cross，支持 macOS (aarch64/x86_64)、Linux x86_64、Windows x86_64 |
| 交互协议 | JNI 1.8 |

### 2. 分层架构设计
```
┌─────────────────────────────────────────┐
│         IDE UI 层 (Kotlin)              │
│  搜索面板 / 结果列表 / 文件预览 / 跳转   │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
│        JNI 适配层 (Kotlin + Rust)       │
│  参数封装 / 类型转换 / 内存管理 / 异常   │
└─────────────────┬───────────────────────┘
                  │
┌─────────────────▼───────────────────────┐
```

### 3. 核心 JNI 接口定义
#### Kotlin 侧 native 声明
```kotlin
package com.example.rustsearch

class RustSearchEngine {
    companion object {
        init {
            System.loadLibrary("rust_search")
        }

        /**
         * 执行文本搜索
         * @param roots 搜索根目录数组
         * @param pattern 搜索模式（字面量或正则）
         * @param isRegex 是否为正则模式
         * @param caseSensitive 大小写敏感
         * @param wholeWords 全字匹配
         * @param includeGlobs 包含文件通配符
         * @param excludeGlobs 排除文件通配符
         * @param contextLines 上下文行数
         * @return 匹配结果数组
         */
        @JvmStatic
        private external fun search(
            roots: Array<String>,
            pattern: String,
            isRegex: Boolean,
            caseSensitive: Boolean,
            wholeWords: Boolean,
            includeGlobs: Array<String>,
            excludeGlobs: Array<String>,
            contextLines: Int
        ): Array<SearchResult>
    }

    data class SearchResult(
        val filePath: String,
        val lineNumber: Int,
        val column: Int,
        val matchedText: String,
        val contextBefore: Array<String>,
        val contextAfter: Array<String>
    )
}
```

#### Rust 侧 JNI 实现
通过 `jni` crate 实现对应 Java 方法，负责参数解析、执行搜索、构建 Java 对象返回。核心是严格管理 JNI 引用，避免内存泄漏。

### 5. 预研阶段具体执行步骤
1. **环境搭建**
   - 基于官方 `intellij-platform-plugin-template` 创建插件项目
   - 创建 Rust 库项目，设置 `crate-type = ["cdylib"]`
   - 配置 Gradle 任务，自动编译 Rust 并拷贝动态库到插件资源目录

   - Rust 侧接入 `ignore` + `grep-searcher`，实现单目录搜索

3. **性能基准测试**

4. **风险点验证**
   - 连续 10 次重复搜索，观察 JVM 内存是否持续上涨（验证是否内存泄漏）
   - 传入中文路径、中文关键词，验证编码正确性
   - 验证大结果集（1 万+ 匹配项）返回的性能

### 6. 核心风险与规避方案
| 风险 | 影响 | 规避方案 |
|------|------|----------|
| JNI 内存泄漏 | 长期使用内存上涨 | Rust 侧使用 `AutoLocal` 自动释放引用，严格遵循 JNI 内存管理规范，加入泄漏检测 |
| 跨平台编译复杂 | 多平台交付困难 | 使用 `cross` 工具做交叉编译，CI 流水线自动构建全平台产物 |
| IntelliJ 版本兼容差 | 部分版本无法使用 | 仅使用公开稳定 API，不依赖内部实现，兼容最近 3 个大版本 |
| 原生扩展点对接复杂 | 开发周期延长 | MVP 先做独立工具窗口，验证价值后再逐步对接原生扩展点 |
